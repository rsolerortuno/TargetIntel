# v0.4.0 target-feasibility contracts

## Scope and boundary

Issue 401 introduces immutable, deterministic, source-linked contracts for
target-feasibility requests, observations, profiles, and validation. They are
research-only explanatory evidence contracts. They create no feasibility
score, target score, rank, biological role, target recommendation, modality
recommendation, go/no-go decision, or clinical conclusion.

This is separate from the v0.1 deterministic prioritization pipeline and the
v0.3 evidence-to-synthesis path. Constructing, validating, or serializing a
feasibility contract does not call or mutate scoring, intent ranking, role
classification, feature-table construction, benchmark evaluation, sensitivity
analysis, LLM extraction/synthesis, human review, `EvidenceItem`, or grounded
synthesis. Offline CI isolation tests protect both non-mutation and
non-invocation.

`targetintel/modality.py` remains the one canonical public modality-evaluation
surface. This package supplies only normalized modality identifiers in
contracts; it is not a modality evaluator and does not replace or alter that
module.

## Controlled contracts

Requests retain an explicit target identifier and identifier type
(`ensembl_gene_id` or `gene_symbol`), optional disease context, requested
dimensions, requested modalities, source/release, and actor identity. No
symbol resolution, target inference, fuzzy matching, synonym expansion,
ontology lookup, automatic selection, environment-derived identity, or default
actor is performed. `gene_symbol` stays unresolved until Issue 402.

Dimensions are descriptive evidence categories: `clinical_precedence`,
`tractability`, `doability`, and `safety`. Modalities are `antibody`,
`small_molecule`, `protac`, and `other_clinical`. They are not weights or
scores. Request order is canonicalized, while duplicates and unknown values
fail closed.

Observations retain target/source-release identity, dimension, optional
modality, factor, typed normalized value, availability state, source record
and source field, deterministic provenance, limitations, and an optional
operational retrieval timestamp. Observations are immutable, source-linked,
and remain separate even when they conflict.

Availability states are `observed`, `not_observed`, `not_available`,
`not_applicable`, `conflicting`, and `retrieval_failed`. `not_available` is
not negative feasibility; `retrieval_failed` is not absence of evidence; and
null remains distinct from false. `not_observed` is meaningful only with the
observation's retained source and record scope.

Safety uses the same availability semantics. In particular, no retrieved
safety signal is not evidence that a target is safe. The contracts reject
`safe`, `risk_free`, `clinically_safe`, and `no_safety_risk` fields.

Profiles canonically order observations and retain count, per-state availability
summary, and contradiction indicators. A contradiction is detected only from
explicit normalized fields: observations sharing target, type, dimension,
modality, factor, and value type but with differing normalized values (or an
explicit `conflicting` state) are retained and flagged without selecting a
scientific winner.

## Deterministic identities and security

Request, observation, and profile IDs are SHA-256 identities over their
respective documented content. Operational timestamps, local paths, process
details, logs, HTTP timings, temporary files, retries, thread IDs, and local
cache locations are excluded. Profiles include canonical observation IDs,
availability summary, contradiction indicators, and their research-only /
no-score / no-ranking / no-role / no-clinical-conclusion boundaries.

Validation is fail-closed for bad identities and vocabularies, provenance,
source-release or target mismatches, duplicate observations, nested mutable or
forbidden content, credentials, hidden reasoning, scoring/ranking/role fields,
clinical-safety claims, and arbitrary filesystem paths. There is no network,
subprocess, dynamic import, unsafe deserialization, raw traceback, or source
document storage in this package.

## Deferred work

Issue 402 will support both association-ranked disease retrieval and directed
retrieval for an explicit target universe. Directed retrieval must support
benchmark targets requested by symbol or stable identifier even if absent from
the top association-ranked results; increasing `page_size` alone is not
sufficient. Its cache identity must include Open Targets release, disease ID,
query type, page size, maximum pages, canonical target-universe hash, and
query/schema version. Equal-sized but different target universes must not
share a cache.

Issue 403 will normalize precedence, tractability, doability, and safety while
preserving per-dimension coverage, missingness, retrieval failures, and
contradictions. It will distinguish no retrieved safety signal from insufficient
safety data and will not produce an aggregate score unless separately approved.
Retrieval-backed, per-dimension feasibility profile construction beyond the
immutable container defined here is therefore deferred to Issue 403.

Issue 404 will extend the existing `targetintel/modality.py` to consume
`TargetFeasibilityProfile` information where compatible. It will not create a
second modality evaluator and will preserve deterministic scoring/ranking
isolation and compatibility tests for existing modality behavior.
